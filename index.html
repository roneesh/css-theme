Text

All apps use text, these are p tags and span tags. Your app should consistently define sizes for various levels of text.

Standard Text:

Standard text,

also Standard <p>Standard text,</p>
<span class="text">also Standard</span>

Small Sized Text:

Small

<p class="text-small">Small</p>

Large Size Text:

Lorem ipsum dolor sit amet

<p class="text-large">Lorem ipsum dolor sit amet</p>

Centered Text:

Lorem ipsum dolor sit amet

<p class="text-centered">Lorem ipsum dolor sit amet</p>

Semantically Important Text:

HTML defines the strong tag, which "indicates that its contents have strong importance, seriousness, or urgency." It also has the em tag, which "marks text that has stress emphasis." In this theme you would need to add the "text-bold" or "text-italic" class to make items appear visually bolded or italicized. There is a minor quirk however, natively browsers style the strong tag as font-weight: "bold", and the em tag as font-style: "italic", so our reset undoes those tags stylings. This allows you to style the unstyled strong and em tags appropriately for the given context.

Important text, bolded
Important text, not bolded

<strong class="text-bold">Important text, bolded</p>
<strong>Important text, not bolded</strong>

Italic text, italicized, Not italicized

<em class="text-italic">Italic text, italicized,</em>,
<em>Not italicized</em>

Semantically Unimportant Text:

HTML defines the small tag, but a small tag semantically means unimportant or text, not small sized text. In this theme you would need to add the "text-small" class. There is a minor quirk however, natively browsers style the small tag as font-size: "smaller", so our reset restores that improper user agent styling to be normal sized.

Small tag, small size
Small tag, normal sized

<small class="text-small">Small tag, small size,</small>
<small>Small tag, normal sized</small>

Links

All apps use links, these are a tags. Your app should not only define styles for links, but also for the three pseudo classes of links: :hover, :focus, :active. Hover over this link to see how it's styling changes:

<a class="link" href="#">Lorem ipsum dolor sit amet</a>

Wrapper Links:

Recently, the HTML standard evolved to allow a tags to wrap any element as a link. In this theme, a tags that wrap block level elements can have a class of "block-link" applied to them. This turns the link into a block level element. This allows layouts to apply vertical margins to the wrapped element through the link. See this example below:

I'm a section wrapped in a link!

 <a class="block-link" href="#">
   <div class="section">...</div>
</a>

Lists

Most every app uses lists. These are <ul> and <ol> tags.

Vertical List:

  • Foo
  • Bar
  • Buz
<ul class="v-list">
   <li>Foo</li>
   <li>Bar</li>
   <li>Buz</li>
</ul>

Horizontal List:

  • Foo
  • Bar
  • Buz
<ul class="h-list">
   <li>Foo</li>
   <li>Bar</li>
   <li>Buz</li>
</ul>

Note that horizontal lists have no list style type.

Ordered Lists:

  1. Foo
  2. Bar
  3. Buz
  1. Foo
  2. Bar
  3. Buz
<ol class="h-list">
   <li>Foo</li>
   <li>Bar</li>
   <li>Buz</li>
</ol>

Ordered lists will have their number in the list in both horizontal and veritcal configurations.

Buttons

Every app uses buttons, and generally there are multiple styles.

Generally, buttons are at least 34px high (and ours are taller), so they can be easily pressed by both a mouse, or a users finger on a mobile device.

<button class="btn btn--primary">
   Submit
</button>

<button class="btn btn--danger">
   Danger
</button>

<button class="btn btn--cancel">
   Cancel
</button>

Finally, it's a common pattern to want a button to look like a link. However it's important to use a tags when a user will leave the page, and use the button tag when the press will trigger an effect on the current page.

Link Button:

<button class="btn btn--link">
   I'm actually a button...
</button>

Tables

Occasionally apps use tables, and they're normally hard to get right. This is a simple, clear table treatment with a responsive design. The responsive design simply collapses rows into blocks, showing each row as a block item. The table utilizes a fixed table layout, with text overflow clipped by ellipses. You can add hover states by adding table--hover.

# Name email Verified? Opted In? On Hold?
1 John Smith John@Smith.com Yes No No
2 Jane Smith Jane@Smith.com Yes No No
<table class="table table--hover">
    <thead>
      <th>Name</th>
      <th>Email</th>
      <th>Verified?</th>
    </thead>
    <tbody>
      <tr>
         <td>John Smith</td>
         <td>John@Smith.com</td>
         <td>Yes</td>
         ...
      </tr>
    </tbody>
</table>

Forms

Forms are the web. Before the ubiquity of JavaScript, forms were how developers made the web custom. Any non-trivial website will have forms. Of all the html elements, forms are where you should strive for semantic, correct markup. Luckily, forms are natively accessible, you just need to use markup as intended.

The markup here is a bit onerous to translate into a code block, so you can right click the form, and click inspect element in your browser.

Form Validations

Forms on the web can accept any input. Permissive inputs is a feature, not a bug about forms. It's best practice to validate both on the server and client side. This theme doesn't implement custom validations and instead relies on HTML5 form validations. They're robust enough to ensure data is formatted properly, and work out of the box natively.
